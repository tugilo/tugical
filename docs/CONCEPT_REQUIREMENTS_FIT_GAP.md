# コンセプト × 要件定義書 フィット＆ギャップ

**作成日**: 2026-02-11 15:43  
**更新日**: 2026-02-11 15:43  

**目的**: 現在のコンセプト（overview・STATUS・.cursorrules に反映済みの思想）と、要件定義書（`tugical_requirements_specification_v1.0.md`）の一致点・ずれを整理し、実装・仕様判断の基準にする。

**参照**:  
- コンセプト元: `tugical_project_overview.md`  
- 要件定義: `tugical_requirements_specification_v1.0.md`（v1.1）

---

## 1. コンセプトの要約（判断の軸）

| 項目 | 内容 |
|------|------|
| **位置づけ** | 「予約を管理するツール」ではなく **「予約の“次の行動”までを設計するツール」**（売上設計ツール） |
| **差別化** | LINE 完結 × 提案型予約 × 売上につながる行動設計 |
| **LINE** | 学習コストゼロ（新しい操作・お客様への説明・アプリ導入が不要） |
| **ペルソナ** | 一次：50代前後・IT苦手・LINEは使える・現場を壊したくない／二次：スタッフ・効率化・提案の仕組み化 |
| **業種** | **初期から業種別に作り込まない**。コアは共通。差は「テンプレ／オプション」で後付け。リリースを止めず導入後に最適化。 |
| **表現** | 「売上が上がる」と断定せず **「売上につながる行動を設計できる」** と表現する。 |

## 2. フィット（一致している点）

要件定義書とコンセプトが噛み合っている部分。**実装・仕様検討の土台**としてそのまま使ってよい。

- **統一概念**  
  - 予約 = リソース × 時間枠 × メニュー（複数組み合わせ含む）。両者で一致。

- **LINE・LIFF**  
  - LINE 連携型、LIFF による顧客識別・予約フロー、10 分仮押さえ。要件定義の顧客側フローと方向性一致。

- **汎用性・時間**  
  - 5 分〜480 分の柔軟な時間スロット、汎用プラットフォームとして多業種を「想定」する記述。コンセプトの「コアは共通」と矛盾しない。

- **提供価値の方向性**  
  - テンプレートで導入しやすく、柔軟な予約条件（性別・設備等）。「提案型」「仕組みが代わりに言う」は要件定義の通知・フロー設計で実現可能な範囲。

- **管理機能の大枠**  
  - 予約管理（カレンダー・Timeline・承認・キャンセル）、メニュー・リソース・顧客・営業時間・LINE 通知。概要レベルで一致。

- **業種を「オプション」で扱う部分**  
  - 要件定義の「業種別カスタマイズ：¥20,000/回」は、コンセプトの「業種は後付けオプション」と整合（有料オプションとして後から提供する想定）。

## 3. ギャップ（ずれ・注意して読む点）

要件定義書をそのまま実装に落とすとコンセプトとぶつかる可能性がある部分。**解釈ルール**を決めておく。

### 3.1 表現・位置づけの差

| 要件定義書の表現 | コンセプトでの扱い |
|------------------|---------------------|
| 「LINE 連携型**予約管理** SaaS」 | コンセプトは **「予約設計」**。管理だけでなく「次の行動」まで設計するツールと位置づける。 |
| 「時間貸しリソース**予約システム**」 | 同じく「設計」のニュアンスで解釈。機能一覧は要件定義のままでも、** product の言い回しは overview に合わせる**。 |

→ 仕様書の文言は「機能」の列挙として尊重しつつ、**製品の打ち出し・優先度は overview を正とする**。

### 3.2 業種の扱い（最重要）

要件定義書には **「業種別○○」** が多数出てくる（業種別表示名、業種別メニュー、業種別必須項目、業種別テンプレート、業種別 KPI、業種別推奨設定など）。

| コンセプト・ルール | 解釈 |
|--------------------|------|
| 業種特化は**現フェーズではやらない**。テンプレは**リリース後にオプション**で追加。 | 要件定義書の「業種別」は **「将来テンプレ・設定で実現する項目」** として読む。 |
| 業種名をロジックに直書きしない。業種差分は設定・テンプレで吸収。 | **実装時は「業種だから特別」という分岐をコードに書かない。** 業種別表示名などは「設定・テンプレで後付け」とみなす。 |

→ 要件定義書の業種記載は **「あるとよい機能のリスト」** であり、**「今すぐ業種別に作り込む」根拠にはしない**。.cursorrules の「業種特化の扱い」を優先する。

### 3.3 ペルソナ・売上・学習コスト

- **ペルソナ**（一次／二次）・**「売上設計ツール」**・**「学習コストゼロ」** は要件定義書にはほぼ書かれていない。  
→ **overview で補っている**。UI 文言・説明文・優先度を決めるときは overview のペルソナと価値表現を基準にする。

- **「最初にラクになること」**（電話削減・次回予約を言わなくてよい等）も overview 側の整理。  
→ 要件定義の機能要件と矛盾はしないが、**優先度・訴求は overview に合わせる**。

### 3.4 スコープ・優先度

- 要件定義書は **機能の網羅リスト** に近い（LIFF 6 ステップ、管理者機能、業種テンプレート、分析・レポートなど）。
- **何をいつやるか** は **STATUS.md と β リリース条件** を優先する。  
  - 例：β では LIFF 単一メニュー完走・仮押さえ・LINE 通知・管理画面での確認が目標。  
  - 業種特化・定期予約・分析は「後付け・拡張」として扱う。

→ 要件定義書の「今後の拡張予定」や Phase 番号は **参考** とし、**リリースを止めない** ことを最優先にする。

## 4. 実装・仕様判断時のルール

1. **コンセプトを最上位に置く**  
   overview の「予約の次の行動を設計する」「売上設計ツール」「業種は後付けテンプレ」を前提にする。

2. **要件定義書の使い方**  
   - 機能の**存在**と**あるべき姿**の参照として使う。  
   - **優先度・スコープ・業種の実装方法** は STATUS.md と .cursorrules（業種特化の扱い）に従う。

3. **矛盾したとき**  
   - 「業種を今やるか／テンプレで後か」→ **後付け**を選ぶ。  
   - 「要件定義に業種別○○とあるが実装でどうするか」→ **設定・テンプレで吸収できる形**にし、業種名の直書きを避ける。  
   - 「機能の範囲」で迷ったら **STATUS.md の完了条件・次にやること** を優先する。

## 5. まとめ

| 観点 | フィット | ギャップ | 判断ルール |
|------|----------|----------|------------|
| 統一概念・LINE・汎用時間 | ◎ 一致 | — | そのまま実装の土台にする |
| 製品の位置づけ・表現 | — | 要件定義は「管理」寄り | 打ち出しは overview に合わせる |
| 業種 | オプション料金などは一致 | 業種別記載が多く「今やる」と読める | 業種は後付けテンプレ。実装では設定で吸収 |
| ペルソナ・売上・学習コスト | — | 要件定義に明示なし | overview を基準にする |
| スコープ・優先度 | 大枠は一致 | 要件定義は網羅的 | STATUS.md と β 条件を優先する |

このドキュメントは、**コンセプトを元に要件定義書とどう付き合うか** を固定するためのものです。仕様の細部は要件定義書を参照しつつ、**何をいつやるか・業種をどう扱うか** はコンセプトと STATUS・.cursorrules で揃えます。
